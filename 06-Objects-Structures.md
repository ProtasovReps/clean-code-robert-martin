# Объекты и структуры данных

> Существует веская причина для ограничения доступа к переменным в программах: мы не хотим, чтобы другие программисты зависели от них.

## Абстракция данных

Создавая абстракции мы вводим правила использования кода. Скрытие реализаций не должно сводится к созданию прослойки функций между переменными, но должно формировать быть направлено на формирование абстракций.

Пример из книги с немного измененным кодстайлом.

```csharp
public class Point
{
    public double X;
    public double Y;
}

// или

public interface IPointable
{
    public double X { get; private set; }
    public double Y { get; private set; }

    public void SetCartesian(double x, double y);
}
```

В примере с интерфейсом помимо абстракции также вводится правило, что устанавливать точки можно только одновременно, в режиме атомарной операции. Второй вариант определяет политику доступа к данным.

Пользователь оперирует уже не с конкретной точкой, а с _сущностью данных_, знать, что конкретно в `IPointable` ему не обязательно.

## Антисимметрия данных/объектов

Дядюшка Боб приводит примеры процедурной и объектно ориентированной реализаций. Процедурный код позволяет легко добавлять новые 
функции без изменения существующих структур данных, но усложняет добавление новых структур данных, потому что требует изменения 
всех функций. В свою очередь, ООП упрощает добавление новых классов без изменения существующих функций, но усложняет добавление новых функций, так как должны измениться все классы.

Простыми словами, написал метод с паттерн матчингом - будь готов добавлять новые матчинги паттернов. Написал базовый класс `Shape` и наследников `Triangle` и `Cube`, решил добавить абстрактный метод `GetPerimeter()` - добавил каждую реализацию к каждому наследнику.

 Иногда процедурный подход бывает обоснованным, главное помнить, что [Яндере Дев тоже любил процедурное программирование](https://www.reddit.com/r/Osana/comments/yzzcmq/yanderedev_code_review/).

 ## Закон Деметры

 Вспомни, что тебе говорила мама: разговаривай с друзьями, не разговаривай с чужаками.

 > Метод `Foo` класса `C` должен ограничиваться вызовом методов следующих объектов:
 >
 > - С;
 > - Объекты, созданные в `Foo`;
 > - Объекты, переданные f в качестве аргумента;
 > - Объекты, хранящиеся в полях класс `C`.
 > 
 > Метод _не должен_ вызывать методы объектов, возвращаемых любыми из разрешенных функций.

С этим законом не все так просто. Например:

```csharp
Azur company = companyPool.Get();

float horizontalPosition = company.transform.position.x;
```

Стоит помнить, что закон эвристический и у него могут быть исключения. Например, `transform` является фундаментальной частью Unity, у него есть своя область видимости и в этом нет ничего плохого. Объекты, содержащие в себе конфигурации, например `playerSettings.Movement.Speed`. `Fluent Builder` по прежнему чувствует себя прекрасно.

## Крушение поезда

Нарушает ли следующий код закон Деметры?

```csharp
Options options = context.GetOptions();
File file = options.GetFile();
string rawData = file.GetRawData();
```

Объем информации для одного метода довольно большой, но все зависит от того, чем является `context`, `options` и `file`. Если они являются объектами (имеется ввиду наличие поведения), то их внутренняя структура должна скрываться, закон нарушается. Если `context`, `options` и `file` являются структурами данных (лишь хранят информацию в себе не имея логики), то они естественным образом раскрывают свою структуру и закон Деметры на них не распространяется.

## Гибриды

Вся эта неразбериха с тем, что можно, что нельзя приводит к появлению так называемых гибридных структур. Это и не объект и не структура данных. Это нечто, собравшее худшее от обоих родителей и кричит о том, что автор не знает, что ему защищать: методы или типы. 

## Скрытие структуры

И все же, если `context` является объектом, что можно сделать? Посмотреть на ситуацию под другим углом! Если он не структура данных, зачем нам брать информацию у него? Для чего? Объект должен выполнять какую-то операцию.

 В том же модуле есть фрагмент кода, который создает новый файл из сырой информации в новом потоке. Почему бы не приказать `context` создавать новый файл. Всей необходимой информацией он обладает и закон Деметры не нарушится.

 ```csharp
context.CreateFileStream(outputStream);
```

## Объекты передачи данных

> Квинтэссенцией структуры данных является класс с открытыми переменными и без функций. Иногда такие структуры называют _объектами передачи данных_ или DTO (Data Transfer Object).

Также есть форма _bean-компонентов_. Разница лишь в том, что в ней поля закрыты и есть доступ на чтение.

```csharp
private readonly string _city;
private readonly int _age;
private readonly Color _skin;

// инит через конструктор
// методы на чтение данных
```

Конечно же я бы этому предпочел автосвойства. Но несколько раз замечал, что Luna с ними может иногда шалить, поэтому будь осторожен.

## Активные записи (Active Records)

Особая разновидность `DTO`, реализуются как обычные или с bean-доступом, но также имеют методы `save` или `find`. При том, что в них есть подобные методы, активные записи все равно интерпретируются как структуры данных, а под бизнес-логику выделяются отдельные объекты.
